> 실행일: 2026-07-05 · 대상: 실제 보도자료 10건 → `run_enrich()` (gpt-4o-mini, LLM 호출 40회)

## 1. 요약 품질 (양호)
- 한 줄 요약은 10건 모두 사실 기반으로 핵심을 잘 포착. `why_it_matters`도 임직원 관점으로 서술됨.
- 라이프사이클 단계도 대체로 타당(의결→DECISION, 후속점검→FOLLOW_UP, 시행중 적금→ENFORCEMENT).

## 3. 명백한 오태깅·오분류 사례와 원인

### E1. `regulation` 에 "정책방안·조치명·회의명"이 법령으로 들어감 (가장 흔함)
- 예: "한시적 금융규제완화조치"(#3), "금융기관 보험대리점 판매비중 규제 개선"(#5),
  "가상자산시장 시세조종 혐의자 수사기관 고발"(#6), "인터넷전문은행 대면업무 범위의 합리적 조정 방안"(#8).
- 원인: 프롬프트가 "법령명"과 "정책/조치/제목"을 구분하지 않음. 모델이 보도자료 제목을 법령으로 취급.
- 영향: `based_on`(근거 법령) 엣지가 오염 → M5 법제처 매핑·라이프사이클 보정 시 매칭 실패.

### E2. `sector` 에 온톨로지 밖 값·기관명 혼입
- 예: "금융투자업권"(#3), "인터넷은행"(#8), "서민금융"(#2·#9), "서민금융진흥원"(#9), "금융위"·"금융위원회"(#4·#10).
- 원인: 시드 온톨로지를 "권장"으로만 제시 → 모델이 자유 생성. 기관/부서와 업권을 혼동.
- 영향: `affects` 엣지가 부정확 → 부서·업권별 발송(그룹 매칭) 정확도 저하.

### E3. 기관 명칭이 구명칭·축약·비존재로 부정확
- 예: "재정경제부"(현 기획재정부), "기획처"(비존재/축약 오류), "노동부"(현 고용노동부), "중기부"·"산업부"(축약).
- 원인: canonical_name 이 거의 항상 null → **정규화가 실제로 작동하지 않음**. 현행 정식명칭 지침 부재.
- 영향: 같은 기관이 다른 표기로 중복 노드화, 그래프 오염.

### E4. `product` 과대추출 / 비상품 혼입
- 예: "생성형 AI"(#5, 상품 아님), "국민성장펀드"·"청년미래적금"·"BNK금융사다리상품"(정책·특정상품명).
- 원인: 상품 정의가 느슨. 기술·정책·고유명을 상품으로 태깅.

### E5. `handled_by` 과다(관계기관 남발) + 소관부서와 미구분
- 예: #1 홈플러스 건에 handled_by 7개 기관(노동부·중기부·산업부·기획처·금융위·금감원·재정경제부).
- 원인: "소관부서(dept)"와 "본문에 언급된 관계기관(agency)"을 구분하지 않음.
- 영향: 이슈-부서 매핑이 넓어져 발송 타깃팅이 흐려짐.

### E6. (경미) importance_score 변별력 약함
- 여러 건이 4.0/5.0 로 동률. 단순 단계가중+영향폭이라 그런 것으로, **설계상 별도 Score 단계(6단계)**에서 정교화 예정.

## 4. 프롬프트/파이프라인 개선안 (다음 단계 반영)

1. **regulation 정의 강화 (E1)**: 프롬프트에 "regulation 은 실제 **법률·시행령·시행규칙·고시·감독규정**의 정식 명칭만.
   보도자료 제목·정책방안·조치명·회의명은 regulation 이 아니다."를 명시하고 positive/negative 예시 추가.
2. **sector/product 를 온톨로지로 제약 (E2·E4)**: "sector 는 아래 목록 중에서만 선택. 목록에 없으면 만들지 말 것.
   기관·부서·정부부처·기술명은 sector/product 가 아니다." → 폐집합(enum) 강제. 필요 시 validation 단에서 온톨로지 밖 sector 는 드롭.
3. **기관 현행 정식명칭 + canonical_name 필수 (E3)**: "정부부처는 현행 정식명칭(예: 기획재정부·고용노동부·중소벤처기업부).
   축약·구명칭 금지. canonical_name 에 표준 명칭을 반드시 채운다." + 약칭→정식 사전 확장.
4. **소관부서와 관계기관 분리 (E5)**: 소관부서는 이미 board.py 가 구조적으로 추출(`press_release.dept`)하므로,
   **handled_by 는 LLM 대신 이 authoritative dept 를 우선 사용**하고 LLM agency 는 보조로만. (구조 데이터 > LLM 환각)
5. **canonical 정규화 실효화 (E3·E2)**: M3 엔터티 매칭 단계에서 동의어 사전으로 정규화·병합(설계서 6.5 정규화 사전).
   LLM 1차 + 사전 2차 보정의 이중화.
6. **regulation enforce_date 활용 (E1 연계)**: M5 법제처 API 로 실제 법령 존재 검증 → 미매칭 regulation 은 플래그.

## 5. 결론
- 요약·단계 분류 품질은 실사용 가능한 수준.
- 태깅(엔터티/엣지)은 **온톨로지 제약 부재 + canonical 미작동**이 핵심 결함. 위 1~4는 프롬프트/파이프라인으로,
  5~6은 M3/M5 사전·API 연계로 개선. 이 결함들은 발송 타깃팅(M4)·라이프사이클(M5)에 직결되므로 우선순위 높음.
